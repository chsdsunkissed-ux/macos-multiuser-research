0. 一句话核心答案

您的问题 一句话答案
A1 保密性：用户二能否看到用户一的文件？ 看不到。~/ 默认 drwxr-xr-x+（POSIX 0755 + ACL），目录下内容默认 700/600，非所有者没有 traverse 进内部的权限。
A2 钥匙串 / 应用数据 / iCloud 是否隔离？ 全部隔离。login.keychain-db 位于 ~/Library/Keychains/（0600），每用户独立 user keybag 由该用户密码 + UID + xART 包裹，跨用户不可解密。TCC 数据库同样每用户独立（~/Library/Application Support/com.apple.TCC/TCC.db）。
A3 管理员能否跨用户访问？ 可以 sudo 到 root（POSIX ACL 全失效），但跑不过 FileVault——VEK（卷加密密钥）由各用户 Secure Token 独立包裹，root 在未解锁用户 A 的 FileVault 时物理上无法读出密文。
B1 快速用户切换：用户二登录后，用户一程序还跑吗？ 继续跑，但 CPU/网/盘 完全可用——只是 GUI 显示、键鼠输入、音频输出切给用户二（用户一进程感知上"像显示器睡着了"）。App Nap 会判定用户一应用非前台、节流其 timer 和 I/O。
B2 登出后登录：用户一程序还跑吗？ 不再跑。GUI 应用：45 秒宽限 → SIGKILL；CLI 进程：SIGHUP → SIGTERM/SIGKILL；用户级 LaunchAgent：SIGTERM → ExitTimeOut 后 SIGKILL。LaunchDaemon 完全不受影响——只跟随系统关机。
决定性分水岭：FUS vs 登出 这是两条完全不同的路径，FUS 下用户一会话只是"切到后台"；登出下用户一会话完全销毁，所有进程接收终止信号。

1. 架构基线（理解后面所有结论必须先懂的事）

1.1 macOS 的三层引导命名空间（Bootstrap Namespaces）

macOS 内核只有一个 PID 1（launchd），但 launchd 维护三层 Mach bootstrap（启动引导）命名空间（来自 Apple TN2083 Bootstrap Namespaces）：

  1. Global namespace（全局命名空间）：root launchd 持有，所有 LaunchDaemon 注册在此
  2. Per-user namespace（每用户命名空间）：每个登录用户的 launchd 实例持有，该用户的 ~/Library/LaunchAgents、/Library/LaunchAgents 注册在此
  3. Per-session namespace（每会话命名空间）：每个 GUI 登录会话 / 每个 SSH 会话独占，Login Items、GUI 应用注册在此

这一架构是理解下面所有行为的基础——快速用户切换下三层共存；登出后 per-user 和 per-session 都被销毁，global 保留。

1.2 macOS 没有 Linux cgroup

macOS 没有 Linux cgroups / systemd slices 的等价物。最接近的是： - ProcessType plist key（系统级内核调度类） - _ResourceCoalition（每应用的资源 coalition，由 RunningBoard 跟踪） - Per-session bootstrap 命名空间（通信隔离，但不限制 CPU/内存）

来自 launchd.plist(5) 文档。

2. 场景 A 详细：多用户的保密性与安全性

2.1 文件与目录隔离

/Users/<username>/ 本身的权限：

drwxr-xr-x+  hoakley  staff
 0: group:everyone deny delete
  • POSIX：0755（owner 全部、group + other 只有 r-x）
  • ACL：0: group:everyone deny delete（防 world 删除目录本身）
  • 目录内部内容默认 700（目录）/ 600（文件）——这是关键的隔离来源

实测命令（来自 Howard Oakley Permissions and ACLs）：

ls -lde ~                          # POSIX + ACL
ls -le /Users                      # /Users 自身
ls -le /Users/<other>/Documents    # 其他用户文档
diskutil info / | grep "File System"
diskutil apfs list

其他用户能否读 ~/Documents / ~/Desktop / ~/Library？

不能。三层防御：(1) ~/ 本身的 o+x 仅允许 traverse 到该目录但看不到内容；(2) 子目录默认 0700；(3) 沙盒应用即使能 traverse 也受 TCC 限制。

关于"~/ 是 0700 吗"的精确说法：/ 自身不是 0700，但其内容默认 0700/0600——这等价于"非所有者进不去"，实际效果一致。来源：Howard Oakley 2022-02-16 Permissions and ACLs + Apple Platform Security Guide § Protecting app access to user data。

2.2 钥匙串（Keychain）隔离

层次 机制 来源
文件系统层 ~/Library/Keychains/login.keychain-db 权限 0600，owner = 该用户 Howard Oakley Keychain History
加密层 每条钥匙串项用 AES-256-GCM 双密钥加密：metadata key（应用处理器缓存）+ row secret key（每次都经 Secure Enclave 派生） Apple Platform Security Guide
用户密钥袋层 ~/Library/Keychains/<UUID>/user.kb 由 (用户密码 + 硬件 UID + xART) 包裹 Apple Platform Security Guide § User keybag

跨用户访问结论：完全不可能。即便用户 B 用 root 权限也读不出密文——密钥袋是用户 A 密码派生的，没 A 密码就是 0 字节随机数。

数据保护类（来自 Apple Platform Security Guide § Data Protection classes）：

保护类 Keychain 等价 多用户行为
Class A NSFileProtectionComplete kSecAttrAccessibleWhenUnlocked "最后一个用户登出后，class key 被丢弃"
Class B NSFileProtectionCompleteUnlessOpen Curve25519 ECDH "只要系统有任何用户登录即可用"
Class C NSFileProtectionCompleteUntilFirstUserAuthentication kSecAttrAccessibleAfterFirstUnlock 默认；首次认证后整个启动期间不解锁

关键引述：Apple silicon Mac 不支持 Class D——每次"登入登出"都构成安全边界（区别于 iOS 的"锁屏"）。意味着 FUS 下 Class C 数据对两个用户都持续可用；但登出登入会重新建立边界。来源：Apple Platform Security Guide。

2.3 应用沙箱（App Sandbox）与 TCC

沙箱规则（Apple Platform Security Guide § App security in macOS）：

  1. 沙盒应用不能读 ~/Library/Containers/ 之外的应用容器（除非有 application-groups entitlement）
  2. 沙盒应用不能 traverse 进其他用户的 ~（POSIX + ACL 阻止）
  3. 沙盒应用不能读 iCloud Drive / 网络卷 / 可移动卷（除非有 com.apple.security.files.user-selected.* 或 Full Disk Access）
  4. macOS 13.4+ 起，所有第三方 app 都有 Data Vault（内核强制）

TCC 数据库是每用户独立：

~/Library/Application Support/com.apple.TCC/TCC.db          # 当前用户授权
/Library/Application Support/com.apple.unifleetd/              # 系统级覆盖
/Library/Application Support/com.apple.TCC/                    # 托管 Mac

来源：Howard Oakley 2025-11-08 TCC Explainer + Apple Platform Security Guide。用户 B 登录不会继承用户 A 的授权——TCC 表是登录时重新构建的。

iCloud Drive 同样按用户分：缓存位于 ~/Library/Mobile Documents/com~apple~CloudDocs/，POSIX 0700；每个用户独立 Apple ID 独立 token；FUS 下 bird daemon 用第二个用户的 Apple ID 重新认证。

Sequoia 新增：屏幕录制授权会被周期性重新弹窗询问（来自 Howard Oakley 2025-03-12 Privacy Overview）。

2.4 管理员 vs 标准用户 vs Root 权限边界

来自 Apple Support → Add a user or group：

角色 可做 不能做
管理员（Administrator） sudo（需重新认证）、为所有用户装 App、增删用户、改系统设置、跑 csrutil status、用自己密码解锁 FileVault 改安全启动设置（需 recoveryOS）、默认不能读其他用户的文件
标准用户（Standard） 为自己装 App、改个人设置 sudo、增删用户、看别人的文件
仅共享（Sharing Only） 通过 SMB/AFP 远程访问共享文件 本地登录、改设置

Root 的真实能力（这是您最关心的问题）：

  • 没 FileVault 时：root 看得见所有用户的 ~/Library、~/Documents（POSIX 0700 被 UID 0 绕过，ACL 也不挡 root）。但 root 也得受 TCC 限制——TCC 是关于"app consent"的，不绑 UID。
  • 有 FileVault 时：root 看得了 已经解锁的卷上的明文；卷没解锁（用户 A 锁屏或系统冷启动）时 root 物理上解密不了——因为 VEK（卷加密密钥）由用户 A 的 Secure Token KEK（密钥加密密钥）包裹，没有 A 密码就没有 KEK。

2.5 FileVault 与 Secure Token——最关键的安全机制

来自 Apple Platform Security Guide § Using Secure Token：

"A Secure Token is a wrapped version of a key encryption key (KEK) protected by a user's password."

机制详解：

  1. APFS 卷有一个 VEK（Volume Encryption Key）
  2. 每个有 Secure Token 的用户用 (自己的密码 + UID + xART) 包一层 KEK，把这个 KEK 写入 VEK 之上
  3. 没 Secure Token 的用户不能解密卷——因为它的密码派生不出那个 KEK

操作后果（来自 Der Flounder 2018-01-20 实测）：

  • macOS 11+ 第一个登录的用户自动获得 Secure Token；后续由该用户创建的用户也自动获得
  • 命令行（sysadminctl -addUser）或 Active Directory mobile 用户默认没有 Secure Token
  • 没有 Secure Token 的用户在预启动解锁界面看不到自己的账户
  • 用户 B 即使在用户 A 解锁后能登录，冷启动后仅用户 A 能解锁

Apple Support 直接确认（来自 support.apple.com/guide/mac-help/use-filevault）：

"If your Mac has additional users, their information is also encrypted. Users unlock the encrypted disk with their login password. If there's an Enable Users button, you must enter a user's login password before they can unlock the encrypted disk."

检查 Secure Token 的命令：

sudo fdesetup status                  # FileVault 是否启用
sudo fdesetup list                    # 哪些用户被启用
sysadminctl -secureTokenStatus <u>    # 查 Secure Token 状态
diskutil apfs listCryptoUsers /dev/disk1   # macOS 14+ 列出 crypto user

2.6 系统级保护机制在多用户场景下

机制 多用户行为 来源
SIP（System Integrity Protection） 对所有进程生效。Apple 原文："macOS applies this security policy to every process running on the system, regardless of whether that process is running sandboxed or with administrative privileges." Apple Platform Security Guide § Restricting components
Gatekeeper / 公证（Notarization） 按文件评估，不按用户。规则全机一致（可通过 MDM 配置或系统设置调整） Apple Platform Security Guide

3. 场景 B 详细：后台进程在切换/登出时的行为

3.1 FUS（Fast User Switching，快速用户切换）下用户 A 进程的行为

Howard Oakley 原文（来自 Fast User Switching 2023-03-22）：

"When you use FUS, your Mac doesn't log out of the current user account, but additionally logs into the other user account, leaving it with three active accounts: root, the first user, and the second."

所以关键事实是：FUS 下用户 A 的会话继续存在，只是切到后台。

程序类别 FUS 下的行为 来源
GUI 前台应用（用户 A） 继续跑。"Processes still running for the first user will continue, but can't receive local keyboard/mouse/trackpad input, and don't output to the display or audio; as far as those processes are concerned, it's as if the display has gone to sleep. All user input and output is thus connected to the second user." Howard Oakley（原文引用）
CLI / SSH 进程（用户 A） 继续跑。每个 SSH 会话属于用户 A per-user 命名空间下的 per-session 子命名空间。CPU / 网 / 盘 全通。sshd 的 ssh.plist 用 SessionCreate=true。 TN2083
用户级 LaunchAgent（~/Library/LaunchAgents/） 继续在用户 A 的 per-user launchd 下运行，不受 FUS 影响。 TN2083 + Designing Daemons
系统级 LaunchAgent（/Library/LaunchAgents/） 每个用户一份实例。用户 A 那份继续跑；用户 B 登录后用户 B 的副本也会 spawn。 TN2083
LaunchDaemon（/Library/LaunchDaemons、/System/Library/LaunchDaemons） 完全不受 FUS 影响。系统级、root 拥有、不知道哪个用户在前面。 TN2083 + CreatingLaunchdJobs
登录项（Login Items） 仍在用户 A GUI 会话里加载，UI 不可见。 TN2083

资源访问（FUS 下用户 A）：

资源 是否可用 信源
CPU ✅ 完全可用，无文档化的 per-session CPU 配额；只有 ProcessType=Background 或显式 ThrottleInterval 才有节流。 launchd.plist(5)
网络 ✅ 可用；BSD socket 不被 FUS 限制；网络策略由 socketfilterfw（ALF）和 entitlement 管。 TN2083
磁盘 I/O ✅ 可用；走 POSIX credentials（UID/GID），不变。 TN2083
内存 ❌ 不主动回收。用户 A 的常驻集被保留——所以 FUS 比 logout-then-login 内存消耗大。 Howard Oakley
WindowServer / 显示 ❌ 用户 A 的 app 不能画到屏幕——WindowServer 连接被切走。 TN2083
音频 ⚠️ 不确定。CoreAudio HAL 可跨会话混音，但实际取决于 app 的 kAudioDevicePropertyHogMode 策略。Howard Oakley 文章评论中有实测分歧（iTunes/Music 停，YouTube 不停）。 Howard Oakley（评论）

节流机制（FUS 下）：

机制 行为 来源
App Nap 用户 A 的 app 触发 App Nap（因为它不是自己会话的前台 app）。CPU 优先级降、定时器节流、I/O 节流。Apple 原文："an app is a candidate for App Nap if: It isn't the foreground app…" Apple App Nap 文档
DAS-CTS 系统级后台调度器（10.10 Yosemite 起），对系统服务（Time Machine / mediaanalysisd / XProtect Remediator）统一调度，与用户无关。 Howard Oakley
ProcessType=Background launchd plist key；轻量 CPU + I/O 节流。 launchd.plist(5)
ThrottleInterval 默认 10 秒。 launchd.plist(5)
NSProcessInfo thermalState 进程级 API（nominal/fair/serious/critical），由 app 自己节流；不是 per-session。 Apple ProcessInfo 文档

3.2 用户 A 登出后 → 用户 B 登录的进程行为

登出状态机（来自 Howard Oakley loginwindow 2023-03-18）：

LogoutConfirmation → StartGUIAppsQuit → AllGUIAppsQuit → 
StartFinderQuit → FinderQuit → PointOfNoReturn → 
StartUIElementsQuit → StartDockQuit → StartWindowManagerQuit → 
FinalItemsBeforeSessionLogoutd → Startsessionlogoutd → LogoutComplete

正常 ~0.5 秒。

程序类别 用户 A 登出后的命运 来源
GUI 应用（用户 A） applicationShouldTerminate → Apple Event quit → 45 秒宽限 → 仍存活则 SIGKILL。所有未保存数据可能丢失。 Howard Oakley
CLI / SSH 进程（用户 A） Per-session 命名空间销毁 → SIGHUP（默认死，除非 nohup） → SIGTERM/SIGKILL。内存数据全丢。 TN2083 + Unix session-leader 语义
用户级 LaunchAgent（~/Library/LaunchAgents/） SIGTERM → ExitTimeOut 后 SIGKILL。用户 A 的 per-user launchd 退出。 CreatingLaunchdJobs + launchd.plist(5) ExitTimeOut
系统级 LaunchAgent（/Library/LaunchAgents/） 同样在用户 A 会话销毁时被 SIGTERM。用户 B 登录后会在用户 B 的 per-user launchd 下重新 spawn。 TN2083 + CreatingLaunchdJobs
LaunchDaemon 完全存活。只在系统关机时收 SIGTERM。 TN2083 + CreatingLaunchdJobs
登录项 被 loginwindow 按 AppleEvent/kill 流程在 LogoutComplete 之前 kill。 Howard Oakley

Apple 官方原文（CreatingLaunchdJobs）：

"When the user logs out, it sends a SIGTERM signal to all of the user agents that it started." "When the system shuts down, it sends a SIGTERM signal to all of the daemons that it started."

数据丢失窗口：

  • Apple 给 GUI app 45 秒宽限期
  • 超过则硬 kill——期间未保存到磁盘的数据丢失
  • 应用可声明 NSSupportsSuddenTermination 或调用 enableSuddenTermination() —— 表示"我没未保存状态"，系统可直接 kill 不通知
  • 来源：Apple ProcessInfo 文档 § Support Sudden Termination

ExitTimeOut 默认值：

"system-defined. The value zero is interpreted as infinity and should not be used, as it can stall system shutdown forever."

所以 launchd 一定会 SIGKILL，只是不立即。

3.3 launchd 体系详解（plist key 多用户语义）

plist 文件位置与归属（来自 launchd.plist(5) FILES + TN2083）：

路径 类型 加载方 运行空间
/System/Library/LaunchDaemons/ 系统 LaunchDaemon root launchd（PID 1） 全局命名空间
/Library/LaunchDaemons/ 第三方 LaunchDaemon root launchd 全局命名空间
/System/Library/LaunchAgents/ 系统 LaunchAgent per-user launchd 该用户的命名空间
/Library/LaunchAgents/ 第三方 LaunchAgent（所有用户） per-user launchd 每个登录用户各一份
~/Library/LaunchAgents/ 第三方 LaunchAgent（单用户） 该用户的 per-user launchd 仅该用户
App.app/Contents/Library/LaunchAgents/ SMAppService 注册（macOS 13+） per-user launchd 该用户；代码签名保护
/private/var/db/ManagedConfigurationFiles/BackgroundTaskServices/LaunchAgents/ DDM-注册（macOS 15+） Managed Configuration 该用户

FUS 下 vs 登出下 的 launchd 行为对比：

类型 FUS（用户 B 登录，用户 A 不动） 登出（用户 A 主动登出）
LaunchDaemon 不受影响 不受影响（只在关机收 SIGTERM）
用户级 LaunchAgent 继续在用户 A per-user launchd 下运行 SIGTERM → ExitTimeOut 后 SIGKILL
系统级 LaunchAgent 用户 A 那份继续；用户 B 那份会 spawn 用户 A 那份 SIGTERM；用户 B 登录后 spawn
登录项 仍在用户 A GUI 会话 被 loginwindow 按 logout 状态机 kill

关键 plist key（multi-user 角度）：

Key 语义 多用户影响
KeepAlive bool 或 dict；true 表示进程退出时重启。"This key implicitly implies RunAtLoad." 用户 A 登出时该 agent 被 SIGTERM；如果没 opt-in to 优雅退出，KeepAlive 帮不到忙（per-user launchd 整个退出了）
RunAtLoad 加载时启动一次。Apple 自家建议："avoided, as speculative job launches have an adverse effect on system-boot and user-login scenarios." 用户登录时启动；用户登出时 SIGTERM

ProcessType Background / Standard / Adaptive / Interactive；"The system will apply resource limits based on what kind of job it is." 决定内核调度类；与用户无关
ThrottleInterval 默认 10 秒；启动频率上限 与用户无关
ExitTimeOut SIGTERM 与 SIGKILL 间秒数；默认 system-defined 用户登出时的死缓时间
EnablePressuredExit 内存压力下可被 kill 与用户无关
SessionCreate 创建新 audit session（sshd 用） 决定 SSH 会话是否独立

AbandonProcessGroup 设为 true 时不杀同 process group 的进程 与用户无关

3.4 Apple 后台任务政策（macOS 14/15）

重要事实：BGTaskScheduler 是 iOS 专用，macOS 上没有。

macOS 用 launchd + DAS-CTS（Duet Activity Scheduler + Centralized Task Scheduling）做后台任务。BGTaskScheduler（BGAppRefreshTask / BGProcessingTask）在原生 macOS 上不可用，仅限 iOS/iPadOS。来源：Howard Oakley 2025-01-25。

DAS-CTS：10.10 Yosemite 起，Apple 的系统级后台调度器（用于 Time Machine、mediaanalysisd、XProtect Remediator）。Howard Oakley 原文："scoring takes into account battery, CPU, thermal, network, etc.; DAS admits 'conditions are deteriorating' in the log when it refuses to dispatch."

Sequoia 新增：DDM（Declarative Device Management）服务后台任务，plist 和可执行文件在 /private/var/db/ManagedConfigurationFiles/BackgroundTaskServices/ 下，受 tamper-resistant 目录保护。来源：Der Flounder 2025-06-10。

App Nap 完整规则（来自 Apple App Nap 文档）：

  • 候选条件：不是前台 app；最近没在可见窗口更新内容；不可听；没持有 IOKit power / NSProcessInfo assertion
  • 节流内容：CPU 优先级降；定时器节流；I/O 节流
  • 唤醒条件：app 切到前台 / 收到 XPC 消息 / 收到 Apple event

7. 一句话总结

macOS 的多用户隔离默认就到位（POSIX 0700 + ACL + TCC + 加密 keybag + per-user launchd 命名空间）——管理员不 sudo 看不见别用户文件，sudo 到 root 看得见未加密卷但穿不过 FileVault。后台进程的关键分水岭是 FUS（Fast User Switching）vs 登出再登录：FUS 下用户一的所有进程继续跑、CPU/网/盘全通，仅 GUI/键鼠/音频切走；登出后用户一的所有进程被 AppleEvent 通知 / SIGTERM / 45 秒宽限后 SIGKILL，仅 LaunchDaemon（系统级守护进程）完全不受影响。尊主如果想让"用户一跑程序 → 用户二登录 → 程序仍继续跑"——用 FUS，不要让用户一登出。
